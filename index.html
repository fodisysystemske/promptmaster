Why this exists

Agent failures are rarely obvious. They are internally consistent and externally wrong. PROMPTMASTER turns risk into a process: evidence, policy, validation, and receipts.

What PROMPTMASTER is

A versioned, publishable, DevOps‑style system that teaches both the human and the agent how to operate in high‑stakes environments.

Dual‑Stream Live Book

Totem‑First Safety

How it works (the operating cycle)

PROMPTMASTER doesn’t “hope” agents do the right thing. It forces an operating loop with explicit gates.

The loop

  1. Interrogation — verify intent, constraints, data classes, risk tier.
  2. Decomposition — produce a sub‑plan with artifacts, previews, rollback.
  3. Totem Spin — independent evidence + policy verdict: PASS/PAUSE/DENY/ESCALATE.
  4. Validation — dry‑runs, diffs, tests, invariants.
  5. Action — gated tool execution only after PASS.
  6. Post‑verify — compare predicted vs actual and store receipts.

About the creator

PROMPTMASTER’s Totem principle is a creator-originated control idea: a cinematic metaphor converted into an engineering transaction that makes agent systems governable.

Origin: the Totem insight

The movie *Inception* captured a real systems problem: inside a dream, everything can feel coherent while being false. Agentic systems reproduce that failure mode in production—especially when tools, logs, and policies are messy.

PROMPTMASTER turns the metaphor into practice: a Totem Spin requires independent legs, explicit verdicts, and post‑verification receipts. That’s why it scales beyond “better prompts” and into operational safety.

FAQ

Short answers that reduce buyer friction.

Is PROMPTMASTER just a prompt pack?

No. It’s a dual‑stream operating doctrine: human governance + agent execution discipline + receipts. Prompts are included as protocols, but the value is the engineered structure and repeatable drills.

Can this guarantee my agent won’t make mistakes?

No credible system can guarantee zero error from stochastic models. PROMPTMASTER makes errors harder to execute by requiring evidence, policy, validation, and explicit escalation.

Why start with the wedge instead of the full 10 kernels?

Because it solves the urgent adoption blockers first: tool safety, auditable receipts, and cost breakers. Expansion packs add depth once the organization trusts the operating loop.
